# スキル実例集

さまざまなユースケースに対応したスキルの実例です。

---

## 例1: コードレビュースキル

**ユースケース**: PRやコード変更のレビューを自動化

```yaml
---
name: code-review
description: コードの品質をレビューし、バグ、セキュリティ問題、改善点を報告する。「レビューして」「コードチェック」「PRを確認」と言われたときに発動。
---

# コードレビュー

## 概要
指定されたコードやファイルに対して品質レビューを実施します。

## レビュー観点

### 1. バグ・エラー
- 明らかなバグや実行時エラーの可能性
- エッジケースの考慮漏れ
- null/undefined チェックの欠如

### 2. セキュリティ
- SQLインジェクション
- XSS脆弱性
- 機密情報のハードコーディング

### 3. 可読性・保守性
- 命名規則の一貫性
- 関数の複雑度
- コメントの適切さ

### 4. パフォーマンス
- N+1クエリ
- 不要なループ
- メモリリーク

## 出力フォーマット

## レビュー結果

### 重要度: 高
- [ファイル名:行番号] 問題の説明
  - 修正提案

### 重要度: 中
- ...

### 重要度: 低（改善提案）
- ...

### 良い点
- ...
```

## 例2: ドキュメント生成スキル

**ユースケース**: コードからドキュメントを自動生成

**構成**: マルチファイル

```
doc-generator/
├── SKILL.md
└── templates.md
```

**SKILL.md**:
```yaml
---
name: doc-generator
description: ソースコードからAPIドキュメント、README、使用例を自動生成する。「ドキュメント作成」「README生成」「API仕様出力」と言われたときに発動。
allowed-tools: Read, Grep, Glob, Write
---

# ドキュメント生成

## 概要
コードを解析し、適切なドキュメントを生成します。

## 対応ドキュメント

1. **README.md**: プロジェクト概要、セットアップ手順、使用方法
2. **API.md**: 公開関数/クラスのリファレンス
3. **CONTRIBUTING.md**: コントリビューションガイド

## 生成ワークフロー

### ステップ1: コード解析
- エントリーポイントの特定
- 公開API（export）の抽出
- 既存ドキュメントの確認

### ステップ2: テンプレート選択
[templates.md](templates.md) からプロジェクトに適したテンプレートを選択

### ステップ3: 内容生成
- コードから情報を抽出
- テンプレートに沿って構成

### ステップ4: 出力
ユーザーに確認を求めてから書き込み
```

## 例3: 設定ファイル検証スキル

**ユースケース**: JSON/YAML/TOML設定ファイルの検証

```yaml
---
name: config-validator
description: 設定ファイル（JSON, YAML, TOML）の構文と内容を検証する。「設定を確認」「configをチェック」「バリデーション」と言われたときに発動。
allowed-tools: Read, Bash
---

# 設定ファイル検証

## 概要
設定ファイルの構文エラーと論理的な問題を検出します。

## 対応フォーマット
- JSON
- YAML
- TOML

## 検証手順

### ステップ1: ファイル特定
対象ファイルを確認（拡張子または内容で判定）

### ステップ2: 構文チェック
- パース可能か
- 正しいフォーマットか

### ステップ3: 論理チェック
- 必須フィールドの存在
- 型の一致
- 重複キーの警告

## 出力フォーマット

## 検証結果: {ファイル名}

### ステータス: OK / NG

### 問題点（ある場合）
- [行番号] 問題の説明

### 警告（ある場合）
- 重複キー、未使用設定など

### 推奨事項
- ベストプラクティスに基づく提案
```

## 例4: 日報作成支援スキル

**ユースケース**: 日報や作業報告書の作成を支援

```yaml
---
name: daily-report-helper
description: 日報や作業報告書の作成を支援する。「日報を書いて」「作業報告」「今日のまとめ」と言われたときに発動。
---

# 日報作成支援

## 概要
今日の作業内容をヒアリングし、日報形式にまとめます。

## ヒアリング項目

1. **完了したタスク**: 今日終わらせた作業
2. **進行中のタスク**: 継続中の作業と進捗
3. **ブロッカー**: 困っていること、相談事項
4. **明日の予定**: 翌日取り組む予定のこと
5. **その他**: 共有事項、気づき

## 出力フォーマット

# 日報 - YYYY/MM/DD

## 完了タスク
- タスク1
- タスク2

## 進行中タスク
| タスク | 進捗 | 期限 |
|--------|------|------|
| タスクA | 70% | MM/DD |

## ブロッカー・相談事項
- （なければ「特になし」）

## 明日の予定
- 予定1
- 予定2

## その他
- 共有事項など
```

## 例5: テスト生成スキル

**ユースケース**: 関数やクラスのユニットテストを自動生成

```yaml
---
name: test-generator
description: 指定された関数やクラスに対するユニットテストを生成する。「テスト作成」「テストを書いて」「テストコード生成」と言われたときに発動。
allowed-tools: Read, Grep, Glob, Write
---

# テスト生成

## 概要
既存コードを解析し、適切なユニットテストを生成します。

## 対応フレームワーク

| 言語 | フレームワーク |
|------|---------------|
| JavaScript/TypeScript | Jest, Vitest |
| Python | pytest |
| Go | testing |
| Rust | cargo test |

## 生成手順

### ステップ1: 対象コードの解析
- 関数/クラスのシグネチャ
- 入出力の型
- 依存関係

### ステップ2: テストケース設計
- 正常系
- 異常系（エラーケース）
- 境界値

### ステップ3: テストコード生成
- プロジェクトの既存テストパターンに合わせる
- モック/スタブの設定

### ステップ4: 確認
生成したテストをユーザーに提示し、承認を得てから書き込み

## 出力例

describe('functionName', () => {
  it('should return expected value for valid input', () => {
    expect(functionName(input)).toBe(expected);
  });

  it('should throw error for invalid input', () => {
    expect(() => functionName(invalid)).toThrow();
  });
});
```

---

## スキル設計のコツまとめ

| ポイント | 説明 |
|---------|------|
| description は具体的に | 曖昧な表現を避け、トリガーキーワードを含める |
| 手順は明確に | ステップバイステップで記述 |
| 出力フォーマットを示す | 期待する形式を明示 |
| 500行以内 | 超える場合は分割 |
| 一貫した用語 | 同じ概念には同じ言葉を使う |
