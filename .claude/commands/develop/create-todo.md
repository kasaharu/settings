# TODO Generator

現在のセッションで議論してきた内容や plan を入力に、開発タスクの TODO を作成し `TODO-*.md` として出力するコマンドです。
事前に課題や方針を会話で詰めてから本コマンドを実行することを想定しています。
この処理のやりとりは日本語でお願いします。

## 入力

このコマンドは **引数を取らない**。入力ソースは以下:

1. **現在のセッションで語られた内容**: ユーザーとのこれまでのやりとりから、解決しようとしている課題・要件・制約・決定事項を抽出する
2. **セッション内で作成された plan**: ExitPlanMode で確定した plan があれば、それを最優先のタスク定義として扱う

両方ある場合は plan を骨格として、会話で補足された詳細（背景・制約・後から決まった方針）をマージする。
plan がない場合は会話内容だけからタスクを抽出する。

### 入力抽出の指針

- **what**: 何を作る・直す・変えるのか
- **why**: その背景・目的（コミットメッセージや概要に使う）
- **scope**: 含めるもの / 含めないもの（会話中で「今回はやらない」と決めたもの）
- **constraints**: 技術的・設計上の制約、ユーザーの好み（例: 「テストは別 PR」「既存 API を壊さない」）
- **decisions**: 会話の途中で確定した設計判断（途中で覆された案は除く）

## タスク

### ステップ 1: コンテキストの自動収集（質問なし）

以下を並列で収集する:

1. **ブランチ情報**: `git branch --show-current` でブランチ名を取得し、チケット番号（例: `MG-7387`）やタスクタイプ（`feature/` → 新規機能、`fix/` → バグ修正、`refactor/` → リファクタリング）を抽出
2. **プロジェクト情報**: `AGENTS.md` / `CLAUDE.md` / `package.json` 等からプロジェクトの技術スタック・規約を把握
3. **既存 TODO**: カレントディレクトリの `TODO-*.md` を確認し、重複があれば警告

### ステップ 2: コード調査・分析

セッションから抽出したタスク定義とブランチ情報をもとに、Agent ツール（Explore）を使って関連コードを深く調査する。
ただし **セッション中で既に調査済みのファイル・確認済みの実装は再調査しない**（重複作業の回避）。

- セッション中に未確認のキーワード・領域について関連ファイルを探索
- 対象ファイルを実際に読み込み、現状の実装を理解する
- 既存のコードパターン・アーキテクチャを把握する
- タスク完了に必要な変更箇所を具体的に特定する

### ステップ 3: TODO の生成・出力

コード調査の結果に基づき、**具体的で実行可能な** TODO を生成してファイルに書き出す。

#### ファイル名

- チケット番号がある場合: `TODO-{チケット番号}.md` (例: `TODO-MG-7814.md`)
- 同一チケットで複数 TODO を作る場合は接尾辞でサブテーマを表現: `TODO-{チケット番号}-{サブテーマ}.md` (例: `TODO-MG-7619-show-translation-page.md`)
- チケット番号がない場合: `TODO-{タスクのキーワード}.md` または日付スタンプ `TODO-YYYYMMDD.md`

#### 出力フォーマット

下記はテンプレートであり、**該当しないセクションは省略してよい** (この方針は後述の「セクションの取捨選択」を参照)。
逆に、セッション内容から必要だと判断した独自セクションを追加してもよい。

```markdown

> Jira: {URL}
> ブランチ: `{ブランチ名}`

> 参考 TODO: `{関連する過去の TODO 相対パス、あれば}`

#### セクションの取捨選択

- **必須**: タイトル・冒頭引用ブロック・目的/ゴール・スコープ・実装ステップ・やらないこと
- **強く推奨**: 進捗 / 変更対象ファイル / コミットメッセージ案 / リスク・未確定事項
- **任意**: 環境セットアップ / 関連コード / ユーザー判断事項 / 関連リンク / 進捗ログ
- 短い修正タスクの場合はテンプレートを大幅に削ってよい (1 ファイル修正なら `目的・ゴール / 変更対象ファイル / 実装ステップ / やらないこと` 程度でも可)

#### TODO の品質基準

- **具体性**: 「テストを書く」「スタイリング」のような汎用語ではなく、ファイルパス・関数名・行番号・キーワードで grep できる程度の具体度
- **why の保存**: 会話の中で出た「なぜそうするか」「なぜそうしないか」を「背景」「方針」「やらないこと」に必ず転記する (会話履歴より TODO のほうが後から見返される)
- **コードの引用**: 重要な変更は diff / 設定ファイルの最終形 / コマンド例をコードブロックで埋め込む
- **検証可能なゴール**: 「完了したか」を自分で判定できるコマンドや確認手順をゴール / 仕上げセクションに含める
- **再開可能性**: 別セッションで再開しても迷子にならないよう、ブランチ名・関連ファイル・前提コマンド (dev サーバ起動など) を冒頭付近に集約する
- **依存関係**: ステップの順序は依存関係を考慮 (設定差し替え → 機械置換 → 手作業 → 検証)
- **行番号は変動する旨を明示**: 行番号を書いた箇所には「行番号は変動しうる。`xxx` で grep が確実」と添える

#### コミット粒度の指針

- **各 Step (`### Step N` / `### Phase N`) が 1 コミットに対応する** ことを意識して分割する
- Step の分割基準: 「この Step だけをコミットしてもビルド / 型チェックが壊れない」状態を保てること
- 変更量が大きい Step は、サブステップ (例: `2-1`, `2-2`) に分割し、それぞれが独立してコミット可能になるようにする
- 各 Step には **コミットメッセージ案** を必ず添える (Conventional Commits + 日本語、et-frontend 等の社内規約に準拠)
- 1 つの Step が 2〜3 コミットになりそうなら、テーブルでコミット粒度の目安を示すこと:

  ```markdown
  | Step | コミット数 | 区切り方 |
  |---|---|---|
  | Step 1 | 1 | 設定変更をまとめて |
  | Step 2 | 2〜3 | ディレクトリ単位で分割 |
  ```

- AI が `git commit` を実行するかどうかは TODO の冒頭付近「作業フロー・運用ルール」に明示する (デフォルトはユーザーが自分でコミット)

### ステップ 4: セルフレビュー

生成した TODO ファイルを、**「新規セッション（このファイル以外のコンテキストを一切持たない状態）で、この TODO だけを読んで開発を完遂できるか」** の観点で必ずレビューする。
ステップ 3 までのコンテキスト（会話履歴・調査結果）は新規セッションには引き継がれないことを前提に、TODO 単体での自己完結性を厳しくチェックする。

#### レビューチェックリスト

以下を全項目チェックする。1 つでも欠けていれば「不足」とみなす。

1. **タスクの所在が分かるか**
   - [ ] チケット番号・Jira URL・ブランチ名が冒頭で確認できるか
   - [ ] 関連する過去 PR / 参考 TODO / 計画ファイルへのリンクが揃っているか
   - [ ] 作業対象アプリ / モジュール名が一目で分かるか

2. **「何を作るか」が分かるか**
   - [ ] 目的・ゴールが、新規セッションの読み手にも具体的に伝わるか
   - [ ] **完了条件**が検証可能なコマンドや観測可能な状態として書かれているか（曖昧な「テストが通る」ではなく具体的に）
   - [ ] スコープ（含める / 含めない）が会話の決定事項を反映しているか

3. **「なぜそうするか」が分かるか**
   - [ ] 背景・why が、TODO 単体で読んでも理解できる粒度で書かれているか
   - [ ] 設計判断（採択した方針 / 却下した代替案）の理由が残っているか
   - [ ] 「やらないこと」の理由・別タスクへの切り出し先が明示されているか

4. **「どこを触るか」が分かるか**
   - [ ] 変更対象ファイルが具体的なパスで列挙されているか
   - [ ] 関連コード（読み取り専用）への参照が、grep キー or 行番号付きで書かれているか
   - [ ] 行番号は変動しうる旨と、行を特定するための grep キーが添えられているか

5. **「どう触るか」が分かるか**
   - [ ] 各ステップが diff / コードブロック / コマンドで具体化されているか
   - [ ] 既存実装のパターン・命名規則を流用する場合、参照先のファイルパスが明示されているか
   - [ ] ステップの依存関係・順序が論理的か

6. **「動いたかどうか」を判定できるか**
   - [ ] 検証コマンド（lint / 型チェック / テスト / dev サーバ起動）が具体的に書かれているか
   - [ ] 検証 URL や mock データの場所が必要なら記載されているか
   - [ ] 失敗した時のトラブルシューティング指針があるか（既知の罠があれば）

7. **再開可能性**
   - [ ] 環境セットアップ（dev サーバ起動コマンドなど）が冒頭で確認できるか
   - [ ] 進捗ログのひな形があり、どこまで進んだかを追記できる構造になっているか
   - [ ] 次に着手すべきアクションが明確か

8. **コミット運用が分かるか**
   - [ ] 各ステップにコミットメッセージ案があるか
   - [ ] コミット粒度の指針（誰がコミットするか、1 コミット / 複数コミットか）が明示されているか

#### 不足が見つかった場合

- **自動補修可能**: ステップ 1〜3 で収集済みの情報や、追加調査で埋められるものは TODO に追記して解消する（必要なら Explore agent を再度起動）
- **判断が必要 / 情報不足**: ユーザーに確認しないと埋められない項目（例: 「テストは別 PR にするか」「PR レビュアーは誰か」）は **TODO 内に明示的に "⚠️ 要確認" マーカー付きで残す**。勝手に推測して埋めない
- **会話履歴依存の表現を排除**: 「先ほど話したように」「あなたが言っていた」のような会話前提の表現があれば書き直す

#### レビュー記録

レビュー結果は完了報告に含める。修正前後で改善した項目と、要確認として残した項目を明示する。

### ステップ 5: 完了報告

以下を報告する:

1. 作成したファイルのパス
2. TODO の項目数（チェックボックス総数）
3. **セルフレビュー結果**:
   - 通過したチェック項目数 / 補修した項目 / `⚠️ 要確認` として残した項目
4. セッションから抽出したタスク定義のうち **TODO に反映しなかった項目**（スコープ外と判断したもの・未決事項）があれば明示する
